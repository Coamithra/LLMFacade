# Codebase Review - Action Items

Critical review of LLMFacade as of the initial commit. Tackle top-down within each section.

## Real bugs

- [x] **#1 Google tool-result roundtrip is broken.** `providers/google.py:188-200` sets `function_response.name = b.tool_use_id`, but `tool_use_id` is a synthetic ID. Gemini matches `function_response.name` against the original `function_call.name`. Fix: store the function name on the `ToolResultBlock` (or look it up from history) and emit it here. Add a Gemini integration test that does a tool roundtrip.

- [x] **#2 Tool-dispatch loop has no max iteration guard.** `conversation.py:207-223` (and `aComplete` mirror at 252-268) is `while True:`. A misbehaving model can spin forever and burn tokens. Add `max_tool_iterations` (default ~16) and raise on overflow. *(Resolved more thoroughly by #3: the loop was removed from Conversation entirely. The cap now lives in `helpers.run_to_completion`, which still raises `ToolIterationLimitError`.)*

- [x] **#3 Streaming silently drops tool dispatch.** Resolved by removing auto-dispatch from `Conversation` entirely. `Complete`/`aComplete` were renamed to `Send`/`aSend` and are now strict single round-trips matching `Stream`/`aStream`. Tool execution moved to `llmfacade.helpers` (`run_bound_tools`, `run_to_completion`, plus async equivalents), built on the public API. Wire-format invariant is now enforced: `Send`/`Stream` raise `ConversationStateError` if any `tool_use` in history lacks a matching `tool_result`. This fixes the silent-corruption failure mode and removes the streaming/non-streaming asymmetry at its root rather than papering over it.

- [x] **#4 `_log_request` writes the entire history every turn.** Resolved by switching to delta logging *and* moving the cache-waste diagnostic to the response side, where the provider's actual `cache_read_tokens` / `cache_creation_tokens` give ground truth. Request records now just carry `new_messages` (delta since last log) plus an abbreviated `prior_history` preview (head/...elision.../tail) — no more local LCP tracking, no `_last_cached_msgs` state. Response records gain a `cache_summary` block computed from `Usage`: `cache_read_tokens`, `cache_creation_tokens`, `uncached_input_tokens`, `hit_ratio`, `approximate_messages_cached` (mapped back via per-provider tokenizer — tiktoken for OpenAI, chars/4 fallback for Anthropic/Google), `tokenizer` label, and a context-aware `_note` (caching working / first-cacheable turn / markers off / TTL-or-divergence / not supported). Per-turn log size is bounded; total log is O(N). Clone treats inherited history as already-logged. OpenAI/Google providers now capture cache stats (`prompt_tokens_details.cached_tokens`, `cached_content_token_count`) so the diagnostic works uniformly across providers. The stats-based approach automatically covers TTL expiry, mid-prefix mutation, missing beta headers, etc., without local state — when something invalidates the cache, `cache_read_tokens` drops and the `_note` flags it.

- [x] **#5 `LLM._default` is a process-wide mutable singleton.** `facade.py:17,28-32`. Mutating `LLM.default().api_keys` leaks across tests/libraries. Add `LLM.reset_default()` and a pytest fixture that resets between tests, or drop the singleton.

- [x] **#6 `LLM.log_dir` is dead code.** `facade.py:26`. Stored, never read. Wire it up as the default root for `Conversation.SetLogging`, or remove the field (and the README/CLAUDE.md mentions).

- [x] **#7 `Provider.NewModel(**kwargs)` and `Model.NewConversation(**kwargs)` swallow typos.** `provider.py:104-107`, `model.py:47-50`. Replace with explicit kwargs.

- [x] **#8 `ToolCall._fn` and `Response.raw` participate in equality/hashing.** `models.py:71-75, 99-108`. Add `compare=False` to both fields.

## Architecture (will haunt us)

- [x] **#9 `_complete_raw` has 12 keyword arguments.** Resolved by introducing a frozen `CompletionRequest` dataclass in `provider.py`. The four abstract hooks (`_complete_raw`/`_acomplete_raw`/`_stream_raw`/`_astream_raw`) now take a single `req: CompletionRequest` argument; concrete providers dropped their `**kwargs: Any` punt and read fields off `req` directly (which restored the type-checking the kwargs flattening had silently discarded). `Conversation._call_kwargs` was renamed to `_build_request` and now returns a `CompletionRequest`; the logging path (`_log_request`/`_log_response`/`_cache_summary`/`_estimate_cached_boundary`) takes the dataclass too instead of reaching into a dict by string key. `MockProvider` and the two tests that built kwargs by hand were updated. Sets up a clean path for #10/#11: collapsing the three settings facades will only touch `CompletionRequest` and the read sites, not 16 hook signatures.

- [x] **#10 Three-level `_SettingsFacade` is reimplemented identically.** Resolved by removing the facade entirely. Generation knobs are now constructor kwargs at every layer (`Provider`, `Model`, `Conversation`), stored as immutable `_defaults` dicts. `CompletionRequest` carries a single merged `settings: dict[str, Any]` plus a `settings_source: dict[str, str]` map; providers read knobs directly via `req.settings.get("temperature")`. The "where do I read this from" branching is gone — there's one dict. The silent-drop-at-wrong-scope bug is gone too: each layer's writes are gated by its own `SUPPORTS` partition. *(Closed as part of the architecture cleanup batch with #11/#12/#13/#14.)*

- [x] **#11 Three separate `Settings` enums.** Resolved by collapsing to a single `RUNTIME_KNOBS` frozenset of string knob names (`"temperature"`, `"max_tokens"`, ...) plus the value enums (`EffortLevel`, `OutputFormat`, `EphemeralCacheTTL`). No more `ProviderSettings` / `Settings` / `ConvoSettings` / `AnySetting` — users write kwargs by name (`temperature=0.7`) and `SUPPORTS` is `frozenset[str]`.

- [x] **#12 PascalCase methods on Python classes.** Renamed throughout: `NewProvider`→`new_provider`, `NewModel`→`new_model`, `NewConversation`→`new_conversation`, `Send`/`aSend`/`Stream`/`aStream`→`send`/`asend`/`stream`/`astream`, `AddUserMessage`/`AddAssistantMessage`/`AddToolResult`→`add_user_message`/`add_assistant_message`/`add_tool_result`, `Snapshot`/`Rollback`/`Clone`→`snapshot`/`rollback`/`clone`, `isAvailable`/`getCapabilities`→`is_available`/`get_capabilities`. `AddSystemBlock`/`AddTool`/`SetLogging` removed (now constructor args). `Start()` removed.

- [x] **#13 Per-call override kwargs are a closed enum.** Resolved by erasure. `_PER_CALL_OVERRIDE_KEYS` no longer exists. `send`/`asend`/`stream`/`astream` accept the same 14 generation-knob kwargs as the constructors, all gated by the model's effective `SUPPORTS`. Adding a new knob is two lines (one in `RUNTIME_KNOBS`, one in each typed signature).

- [x] **#14 Two sources of truth for `BaseURL`.** Resolved by removing `BaseURL` from the settings system entirely. `base_url` is now a constructor-only identity arg on `Provider.__init__`, stored as `self._base_url`, used directly in `_init_client()`. There's nowhere to set it post-construction, so the dual-write hazard is gone. `org_id` follows the same pattern (it's a constructor-only arg on `OpenAIProvider`).

- [x] **#15 Anthropic `_NO_THINKING_MODELS` substring match.** First removed the substring heuristic entirely (it only existed to coddle deprecated 3.x callers — the whole current Claude 4.x lineup supports `thinking`). Then replaced it with a proper `AnthropicModel` enum that carries capability metadata per known model id. Final scope: current generation only — `OPUS_4_7`, `SONNET_4_6`, `HAIKU_4_5`. All three carry the full SUPPORTS set; the `(model_id, capabilities)` shape is preserved so a future non-thinking or partially-capable model can be added without touching the call signature. `AnthropicProvider.new_model` accepts `AnthropicModel | str`: enum member resolves both the canonical id and `capability_override` automatically; raw string falls back to full SUPPORTS and the caller is on their own (this is the path for deprecated 3.x models — pass `capability_override=provider.SUPPORTS - {"thinking"}` explicitly). Explicit `capability_override=` always wins over enum default. Three tests in `test_capabilities.py` cover all three paths. Bumped `tests/test_conversation.py` from `claude-sonnet-4-5` to `claude-sonnet-4-6` while in the area. CLAUDE.md updated. The enum is a per-release snapshot — bump it when adding new generations; no rot vector beyond library version boundaries.

- [x] **#16 Tool call IDs synthesized from `id(obj)`.** `google.py:175,245,247`, `ollama.py:190,266`. CPython reuses ids after GC. Switch to `uuid.uuid4().hex` or a per-conversation counter.

- [x] **#17 Streamed assistant turns lose thinking content.** Resolved by adding `ThinkingBlock(text, signature, encrypted, provider_data)` to the wire-format models and persisting it in `Message.content` from both `send` and `stream`. Investigation showed the original framing was incomplete: both paths actually dropped thinking from history (only `Response.thinking` exposed it on the non-stream side), and the deeper concern was Anthropic / Gemini extended-thinking-with-tool-use, which requires the integrity-signed thinking block to be returned verbatim in subsequent turns. The single block type covers all four providers: Anthropic emits `thinking`/`redacted_thinking` (with `signature` or encrypted `data` in `provider_data`); Gemini emits parts with `thought=True` + `thoughtSignature`; OpenAI Chat Completions skips on the way out (Responses API not wired); Ollama skips (local models with no signature requirement). Stream parsers buffer `signature_delta` (Anthropic) or accumulate thought parts across chunks until the next non-thought part (Gemini), then emit a finalized `StreamEvent(thinking_block=...)` event that `_finalize_stream` collects into the assistant `Message.content`. Order is preserved: thinking → text → tool_use, which is what both Anthropic and Gemini require on the way back. `Response.thinking` is kept as a derived convenience string for back-compat. Tests in `tests/test_thinking.py` cover stream/non-stream parity, async streaming parity, Anthropic round-trip with signature and redacted_thinking, the multi-turn thinking+tool_use scenario, and the Gemini converter (including the encrypted-block drop). 12 new tests, 80/80 total passing.

- [x] **#18 `tools.py` falls back to `{"type": "string"}` for unknown annotations.** `tools.py:90-91`. Pydantic/dataclass annotations silently become strings -> model passes a string -> function TypeErrors at call time. Raise at decoration time or warn loudly.

- [x] **#19 Lazy intra-package imports.** `provider.py:48`, `conversation.py:455`, `facade.py:100`. No circular-import risk. Move to module top.

- [x] **#20 Sync and async stream parsers are copy-pasted in every provider.** Original framing was partially out of date by the time this was tackled: OpenAI, Google, and Ollama had already factored their per-chunk parsing into a shared sync `_chunk_to_events(...)` helper during the streaming pass, so their sync/async `_*stream_raw` wrappers are short and only the `try/except` error-mapping is duplicated. Anthropic was the lone hold-out — `_iter_stream_events` and `_aiter_stream_events` were 65-line near-clones differing only in `for`/`async for` and the one `await stream.get_final_message()`. Resolved by applying the same pattern: extracted `_chunk_to_events(event, state)` taking one SDK event plus a state dict (`current_tool`, `current_thinking`), and the sync/async `_*stream_raw` wrappers now iterate the SDK stream and forward each event to the shared helper, handling `message_stop` inline because that's the one place sync/async genuinely differ. Also hoisted `import json as _json` to module top (was inlined inside both stream loops). Net: ~60 lines removed, Anthropic now structurally matches the other three providers. Did not factor the wrapper-level boilerplate (provider-specific exception mappings + sync/async iteration mismatch is a Python language constraint) — abstracting over it would obscure more than it saves.

- [x] **#21 `MockProvider` `@dataclass` + explicit `__init__` does nothing useful.** `tests/conftest.py:29-80`. Drop the decorator.

- [x] **#22 `OpenAIProvider._message_to_api` silently drops images on assistant messages.** `providers/openai.py:274-277`. Round-tripping a vision response loses image blocks. Either raise or warn.

- [x] **#33 Google and Ollama silently drop `tool_choice`.** Resolved by reframing `tool_choice` as a regular runtime knob rather than a special-case field. Two architectural moves: (1) split tool-related capability into two flags — `"tools"` (basic tool calling, "auto" mode) and `"tool_choice"` (forced selection: `"required"` / `"none"` / `<tool_name>`), declared per provider in `SUPPORTS`. A model can advertise `"tools"` without `"tool_choice"`, which is exactly Ollama's situation. (2) Add `"tool_choice"` to `RUNTIME_KNOBS` and remove the dedicated `CompletionRequest.tool_choice` field; the value lives in `req.settings` like every other knob and flows through the standard `provider < model < convo < per_call` cascade. Per-provider declarations: Anthropic / OpenAI / Google have both `"tools"` and `"tool_choice"`; Ollama has only `"tools"` (so any non-default `tool_choice` raises `UnsupportedFeature` at the layer where it was set, no provider-specific code needed). Wired Google's `_tool_choice_to_api` to emit `tool_config = {"function_calling_config": {"mode": "AUTO"|"ANY"|"NONE", "allowed_function_names": [...]}}`. Also fixed an adjacent Anthropic bug: the `"none"` translation was returning `{"type": "auto", "disable_parallel_tool_use": False}` (a no-op — same as `"auto"`); it now returns `{"type": "none"}` matching Anthropic's actual API. Conversation-level validation (in `_build_request`): a non-`"auto"` `tool_choice` requires registered tools, and a named-tool value must match a registered tool's name; either failure raises `ValueError` before the request leaves the facade. Models that don't support tool calling at all use `capability_override` at `new_model()` to drop `"tools"` from their effective SUPPORTS — same pattern as Anthropic 3.x non-thinking models. New `Conversation.__init__` check raises `UnsupportedFeature` when `tools=[...]` is passed to a model without `"tools"` capability. Tests: 29 cases in `tests/test_tool_choice.py` covering all four canonical values × four providers, the cascade, the named-tool validator, the Ollama capability-rejection path, and the per-model tools disable.

- [x] **#32 Real tokenizers for Anthropic & Google `cache_summary` boundary estimate.** Resolved in two parts. (1) **Turn-boundary tracking** sidesteps the tokenizer entirely in the common case: `Conversation` now records `(msg_count_at_send, total_input_tokens)` from each response's `usage` (free — no extra API calls) and `_estimate_cached_boundary` first looks for an exact match in that list before falling back to the per-message tokenizer walk. Provider cache markers always sit at turn boundaries (system blocks + a previous turn's last user message), so a `cache_read_tokens` value typically equals some prior turn's recorded total exactly — yielding an exact message-index answer. The `tokenizer` field in `cache_summary` reports `"exact (turn-boundary)"` when matched. Snapshot/rollback/clone all carry the boundary list correctly. (2) **Public `count_tokens` API.** Promoted the per-provider tokenizer hook from `_estimate_tokens` (private) to `Provider.count_tokens(text, *, model_id=None)` plus `Provider.tokenizer_name(model_id=None)`, with ergonomic `Model.count_tokens(text)` / `Model.tokenizer_name()` wrappers. Wired up Google's `LocalTokenizer` (sentencepiece-backed, `google-genai[local-tokenizer]`) covering Gemini 2.0/2.5/3.0-preview — exact, offline after a one-time `.spiece.model` fetch. Added a `tokenizers` extra in `pyproject.toml` bundling tiktoken + sentencepiece. Anthropic and Ollama keep `chars/4` as the local default (no official offline tokenizer for either; users wanting exact Anthropic counts can call `client.messages.count_tokens` directly via the SDK — kept out of the logging hot path). Tests in `tests/test_token_counting.py` (11 cases) cover the public API, override pattern, turn-boundary record/match/fallback, rollback clip, clone inheritance, and stream-path recording.

## Project Overview

LLMFacade is a Python library providing a unified interface to multiple LLM providers (Anthropic, OpenAI, Google Gemini, llama.cpp via `llama-server`). Zero required runtime dependencies - provider SDKs are lazy-loaded only when used. Python 3.10+.

**Integration tests cost money. Do not run them without express permission.**

`tests/integration/` hits real provider APIs (Anthropic, OpenAI, Google) and burns credits on every run. They are gated behind `-m integration` and skipped by default — keep it that way. Never invoke `pytest -m integration`, `pytest -m "integration or not integration"`, `pytest tests/integration/`, or any variant that includes them, unless the user has explicitly asked for that specific run in the current turn. Past authorization in earlier turns does not carry over. The llamacpp integration test is local and free, but still requires explicit permission so the rule stays simple: never auto-run anything under `tests/integration/`.

## Architecture

Four-level hierarchy. Each level owns its own concerns and spawns the next:

```
LLM            manager: shared api_keys; LLM.default() is a process-wide singleton
 -> Provider   identity (api_key, base_url) + SDK client + generation defaults
   -> Model    a model_id bound to a provider, with optional model-level defaults
     -> Conversation   stateful session: history, system blocks, tools, convo-level defaults
```

**Configuration is constructor-only.** Identity (api_key, base_url, model_id, system_blocks, tools, log_dir, log_path) is supplied at construction and never changes. Generation knobs (temperature, max_tokens, etc.) are accepted as kwargs at every layer and form a cascade (`provider < model < convo < per_call`); they are also immutable post-construction. There is no `Start()` step — conversations are usable immediately after construction.

Key files:

- `src/llmfacade/facade.py` — `LLM` manager; `new_provider(name, **kwargs)` dynamically imports a provider module via `PROVIDER_REGISTRY` and forwards kwargs. Also hosts the `generate_image` / `agenerate_image` single entry point (resolves and caches a provider per `(name, base_url)`; see the **Image generation** section).
- `src/llmfacade/provider.py` — `Provider` base class. Owns API-key resolution (override > manager dict > env var), `_init_client()`, the merged `CompletionRequest`, and the `_complete_raw` / `_acomplete_raw` / `_stream_raw` / `_astream_raw` hooks. Also defines the `SystemBlock` dataclass, the cascade helpers `_validate_knobs` / `_filter_unsupported`, and the image surface. `generate_image` / `agenerate_image` are **concrete template methods**: they call the provider hook `_generate_image_raw` / `_agenerate_image_raw` (base raises `UnsupportedFeature("image_generation", ...)`, so non-image providers fail fast and `is_available("image_generation")` reads False for them), then `_log_image_generation` writes the run-dir audit ledger. `new_image_model` mirrors `new_model`. See the **Image generation** section.
- `src/llmfacade/model.py` — `Model` binds a `model_id` to a `Provider` and stores its own generation defaults. Supports a `capability_override` to narrow the provider's `SUPPORTS` set for a specific model (e.g. when calling a deprecated model that doesn't honor a knob the provider generally supports).
- `src/llmfacade/image.py` — `ImageModel` binds an image `model_id` to a `Provider` with optional per-model defaults (`n` / `size` / `aspect_ratio` / `quality` / `background` / `output_format`); `generate` / `agenerate` forward to the provider's `generate_image`. The image analog of `Model` (image generation is a one-shot, not a `Conversation`). See the **Image generation** section.
- `src/llmfacade/_image_log.py` — the image-generation audit ledger. `resolve_image_log_path` (the `log_dir`-cascade resolver: provider override > manager run-dir), `build_image_record` (the JSONL record), and `log_image_generation` (best-effort append of the JSONL line + an HTML `<section>`). The image analog of `_html_log.py`; see **Image-generation logging** under the **Image generation** section.
- `src/llmfacade/conversation.py` — `Conversation` is the stateful API: history, system blocks (with optional `cache=True`), tools, JSONL logging, `snapshot`/`rollback`/`clone`. `send`/`asend`/`stream`/`astream` are strict single round-trips that accept the full set of generation-knob kwargs as per-call overrides. The wire-format invariant (every `tool_use` matched by a `tool_result`) is enforced at request time and raises `ConversationStateError` if violated.
- `src/llmfacade/helpers.py` — Optional convenience built on the public Conversation API. `run_bound_tools(convo, resp)` dispatches every tool call whose name matches a `@tool` registered on the conversation and appends results. A tool may return an `ImageBlock` (or a block list); the helper embeds it in the tool result when the model declares `"tool_result_images"`, else re-attaches it as a follow-up user message (see **Tool-result images** under Capability gating). `run_to_completion(convo, prompt)` is the agent loop (`send` -> dispatch -> `send` -> ...) with a `max_iterations` cap, raising `ToolIterationLimitError` on overflow. Async equivalents: `arun_bound_tools`, `arun_to_completion`. Helpers touch only the public surface — no underscore attributes.
- `src/llmfacade/settings.py` — `RUNTIME_KNOBS` frozenset (the 16 string knob names) plus the value enums `EffortLevel` (`LOW`, `MEDIUM`, `HIGH`, `XHIGH`, `MAX` — the Anthropic `output_config.effort` values; default `HIGH`, with `XHIGH`/`MAX` Opus-tier only), `ThinkingMode` (`ADAPTIVE`, `ADAPTIVE_SUMMARIZED`, `DISABLED` — adaptive-thinking modes for the `thinking` knob; an `int` value still selects legacy budget-based thinking), `ThinkingStyle` (`TEMPLATE_KWARG`, `REASONING_BUDGET`, `THINK_TOKEN`, `DEFAULT`, `UNKNOWN` — how a local GGUF's chat template gates reasoning, auto-detected by the llamacpp provider; see its **Thinking-style auto-detect**), `OutputFormat`, `EphemeralCacheTTL` (`FIVE_MINUTES`, `ONE_HOUR`), and `DrySampler` (the frozen-dataclass value of the llamacpp-only `dry` knob — see the llamacpp **DRY sampler** quirk). The `is_budget_thinking()` helper classifies a `thinking` value as the budget form (int) vs an adaptive mode. `LAUNCH_KNOBS` also lives here (the llamacpp managed-mode launch knobs).
- `src/llmfacade/models.py` — Frozen dataclasses for the wire format: `Message`, `TextBlock`, `ImageBlock`, `ToolUseBlock`, `ToolResultBlock`, `ThinkingBlock`, `ToolCall`, `Response`, `Usage`, `StreamEvent`. `ImageBlock` has `from_path` / `from_base64`. `ImageResult` (the value `generate_image` returns: a list of `ImageBlock`s, an `ImageUsage`, `model`/`provider`, optional written `paths`, plus a `.save(dir)` helper) and `ImageUsage` are the image-generation result types. `Usage` carries `reasoning_tokens` (a subset of the output the model produced, spent on chain-of-thought): populated from the provider's native usage where it's broken out (OpenAI `completion_tokens_details.reasoning_tokens`, Google `thoughts_token_count`), `0` otherwise (Anthropic, most llama.cpp builds, which fold it into `completion_tokens`).
- `src/llmfacade/tools.py` — `@tool` decorator that builds a JSON schema from a function's signature + type hints + docstring. Handles primitives, `Literal`, `Optional`/`Union`, `list[T]`, `dict`.
- `src/llmfacade/exceptions.py` — Hierarchy rooted at `LLMError`: `AuthenticationError`, `RateLimitError`, `ProviderError`, `ModelNotFoundError`, `ProviderNotInstalledError`, `UnsupportedFeature`, `ToolIterationLimitError`, `ConversationStateError`, `CacheMissError`.
- `src/llmfacade/cache.py` — `ResponseCache` (filesystem backend), `fingerprint_request` / `hash_fingerprint` (canonical hash inputs), `replay_stream` / `areplay_stream` (synthesise a stream from a cached `Response`), and `resolve_cache` (cascade resolver). See the **Response cache** section below for behaviour.
- `src/llmfacade/providers/__init__.py` — `PROVIDER_REGISTRY` mapping names to `(module_path, class_name)`. `"google"` and `"gemini"` both resolve to `GoogleProvider`.
- `src/llmfacade/providers/{anthropic,openai,google,llamacpp}.py` — Provider implementations.

### Settings cascade

Every generation knob can be defaulted at any of four scopes:

1. `Provider(... temperature=0.7)` — applies to every model and convo under this provider.
2. `provider.new_model("...", temperature=0.7)` — overrides the provider default for this model.
3. `model.new_conversation(temperature=0.7)` — overrides for this convo.
4. `convo.send("...", temperature=0.7)` — overrides for this single call.

Precedence on read is `provider < model < convo < per_call` (later wins). Unknown kwarg names raise `TypeError`. Knobs not in the relevant layer's effective `SUPPORTS` raise `UnsupportedFeature` at the layer they're set. If a higher-scope default is for something a downstream model doesn't honor (e.g. `thinking` set on the Anthropic provider but the convo binds to a non-thinking model), the cascade silently drops the value at request time and warns once per `(key, source, model)` tuple.

### Logging

Logging is **on by default**. `LLM(log_dir=..., max_log_folders=10)` configures the manager-level root and retention. Each `LLM` instance reserves a session-stamped subfolder `<log_dir>/llmfacade<YYYYMMDD-HHMMSS>/` (default base: `<cwd>/logs`). On first write, the manager prunes older sibling `llmfacade*` directories down to `max_log_folders` and materialises the new one. Each `Conversation`'s log file is `<run_dir>/<convo.name>.jsonl` with an HTML sibling. The convo's `name` is auto-generated (`convo-<8hex>`) unless you pass `name=` to `new_conversation`.

`log_dir` cascades: convo > model > provider > manager. Any layer can pass `log_dir=False` to disable logging for its scope; a lower layer can re-enable by supplying its own `log_dir`. `Conversation(log_path=Path(...))` is an explicit-file override that bypasses the cascade entirely; `log_path=False` disables logging for that one convo.

The JSONL log starts with a single `settings` header record listing every effective knob, its value, and which scope (`provider`/`model`/`convo`) supplied it — plus the system blocks and tool names. Subsequent entries are tight: `request` records carry only `overrides` (per-call kwargs) and `new_messages` (delta since last log), and `response` records carry the assistant content and a `cache_summary` block.

**Reasoning in the log.** Any `ThinkingBlock` in the assistant turn renders as a default-collapsed `<details>` block in the HTML log (and the response record's `thinking` field in JSONL), so reasoning is visible and unfoldable. The response record also gains a `reasoning` summary block — `{reasoning_tokens, source, estimated, chars}` — and the HTML usage line gains a `reasoning N` badge next to `out`. The token count prefers the provider-reported `usage.reasoning_tokens`; when the provider doesn't break reasoning out (llama.cpp folds it into `completion_tokens`; Anthropic doesn't report it), `Conversation._reasoning_summary` falls back to counting the reasoning text (`Response.thinking`) with the provider's local tokenizer — **exact** for llama.cpp (`/tokenize`), a labelled `chars/4` estimate for Anthropic — and the badge is prefixed `~` and marked `estimated`. The local count runs only when logging is active and only on turns that produced reasoning.

`cache_summary.approximate_messages_cached` maps `cache_read_tokens` back to a message index. The lookup uses **turn-boundary tracking**: each successful send/stream records `(msg_count_at_send, total_input_tokens)` from `usage`, and a later turn's `cache_read_tokens` is matched exactly against that list (cache markers always sit at turn boundaries, so a hit typically equals some prior turn's recorded total). When matched, `tokenizer` reports `"exact (turn-boundary)"`. If no recorded boundary matches (first-turn caching, system-block-only markers, mid-prefix divergence after rollback), it falls back to a per-message walk via `Provider.count_tokens` — exact for OpenAI (tiktoken), Google (sentencepiece via `google-genai[local-tokenizer]`), and llamacpp (server `/tokenize`); `chars/4` for Anthropic.

### Token counting

`Provider.count_tokens(text, *, model_id=None)` and `Provider.tokenizer_name(model_id=None)` are the public local-tokenizer API. Convenience wrappers `Model.count_tokens(text)` and `Model.tokenizer_name()` bind the model id automatically. Always local — never makes an external network call (the llamacpp provider calls its own running `llama-server`'s `/tokenize`, which is local by definition). Install the optional `tokenizers` extra (`pip install llmfacade[tokenizers]`) to enable tiktoken (OpenAI) and sentencepiece (Google). Anthropic has no offline tokenizer and returns `chars/4`; for exact counts call `client.messages.count_tokens` via the SDK directly. llamacpp uses the running server's `/tokenize` and falls back to `chars/4` on connection error.

### Response cache (deterministic replay)

Off by default. Set `cache_dir=<path>` at any of provider, model, or conversation scope to enable a filesystem-backed cache of `Response` objects. On a hit, no provider call is made — the stored response is returned (or replayed as a stream). The hash key covers every input that affects output: provider name, model id, system blocks (including `cache=True` markers — flipping caching gets fresh output, by design), the full message list (image bytes hashed via SHA-256), tool schemas in registration order, the merged effective settings (with `Enum`s normalised to `.value`), and the `stop` list. Storage layout: `<cache_dir>/<provider>/<model_id>/<sha256>.json`, where each file holds the canonical fingerprint (for inspection) plus the serialised response.

`cache_dir` cascades convo > model > provider exactly like `log_dir`. Pass `cache_dir=False` at any scope to disable for that scope; a lower scope can re-enable with its own `cache_dir`. `cache_mode` cascades the same way (default `"read_write"`):

- `"read_write"` — read on hit, call provider on miss and write the result.
- `"read_only"` — read on hit, call provider on miss but do not write.
- `"record_only"` — always call provider, write the result (overwrites any existing entry).
- `"replay_only"` — read on hit, raise `CacheMissError` on miss. No provider call. Use this in CI to guarantee no accidental API spend.

Streaming hits are reconstructed by `cache.replay_stream(resp)`: thinking blocks (and their deltas) come first, then a single `text_delta` carrying the full text, then one event per tool call, then a terminal `done` event with the cached `usage` and `finish_reason`. This is enough to drive any consumer that handles real streams; faithful chunk-timing replay is intentionally not attempted. Hits skip `_finalize_stream` entirely so the cached blocks land in history in their original order.

### Capability gating

Every provider declares `SUPPORTS: frozenset[str]` listing the knob names it accepts. Setting an unsupported knob at any layer raises `UnsupportedFeature`. Use `provider.is_available("temperature")` / `model.is_available(...)` / `convo.is_available(...)` and `get_capabilities()` to query (returns plain string sets). Never catch `UnsupportedFeature` to branch — query first.

`SUPPORTS` also carries pure capability flags that are **not** runtime knobs (not in `RUNTIME_KNOBS`, never settable kwargs): `"tools"`, `"tool_choice"`, `"vision"`, `"tool_result_images"`, `"thinking_budget"`, and `"image_generation"`. They exist purely for `is_available` / `get_capabilities` queries and the matching enforcement gates.

**Vision.** All four stock providers declare `"vision"` and marshal `ImageBlock` content on user messages into their native wire shape (Anthropic `{"type":"image","source":{...}}`, OpenAI/llamacpp `{"type":"image_url","image_url":{"url":"data:..."}}`, Google `{"inline_data":{...}}`). Attach images via a content-block list: `convo.send([TextBlock("describe"), ImageBlock.from_path("x.png")])` (or `add_user_message(content=[...])`). The gate is enforced at **request time** in `Conversation._build_request` (the single funnel for `send`/`asend`/`stream`/`astream`, alongside the wire-format invariant): if any message in history carries an `ImageBlock` (top-level or nested in a `ToolResultBlock`) and the bound model is not `is_available("vision")`, it raises `UnsupportedFeature("vision", ...)` before any provider call. Model-level vision support (e.g. a text-only model, or a llamacpp GGUF launched without `--mmproj`) is expressed by narrowing: `capability_override=provider.SUPPORTS - {"vision"}` on `new_model()`. The `AnthropicModel` enum members are all multimodal and inherit `"vision"` from `_SUPPORTS`.

**Tool-result images (`"tool_result_images"`).** An `ImageBlock` returned by a tool (nested inside a `ToolResultBlock`) is a separate capability from `"vision"`, because only Anthropic's API accepts an image in tool-result position — its `_content_to_api` marshals it; OpenAI/Google/llamacpp have no image slot in a tool/function-result message. Only Anthropic declares `"tool_result_images"`. The same request-time gate enforces it: a nested-in-tool-result `ImageBlock` against a model without `"tool_result_images"` raises `UnsupportedFeature("tool_result_images", ...)` (instead of the image being silently dropped). Top-level images stay gated by `"vision"`; the two are checked independently. The auto-loop helpers (`helpers.run_bound_tools` / `arun_bound_tools`) handle the fallback automatically — a tool may return an `ImageBlock` (or a `[TextBlock, ImageBlock]` list), and when the model lacks `"tool_result_images"` they reduce the tool result to text and re-attach the image(s) as one follow-up user message (gated by `"vision"`) after the batch, instead of letting the gate raise.

**Thinking modes (`"thinking"` knob vs `"thinking_budget"` flag).** The `thinking` knob takes either an `int` (legacy budget-based extended thinking → `{"type":"enabled","budget_tokens":N}`) or a `ThinkingMode` / its string value (`ADAPTIVE` → `{"type":"adaptive"}`, `ADAPTIVE_SUMMARIZED` → adds `display:"summarized"`, `DISABLED` → `{"type":"disabled"}`). Anthropic's `AnthropicProvider._thinking_to_api` performs the mapping. These are two API forms with different model support: Opus 4.7/4.8 accept adaptive thinking but **400 on a budget** (sampling knobs are gone too — `effort` controls depth). The split is expressed with two capabilities: `"thinking"` (the knob is settable, gated like any knob name in `SUPPORTS`) and the pure flag `"thinking_budget"` (the budget *form* is accepted). `_OPUS_4_8_SUPPORTS` keeps `"thinking"` but drops `"thinking_budget"`, so adaptive thinking works on Opus 4.8 while a budget value fails fast. The budget gate is **value-level**, so it lives in `Conversation._build_request` (alongside the vision gate): if the merged `thinking` value `is_budget_thinking(...)` and the model is not `is_available("thinking_budget")`, it raises `UnsupportedFeature("thinking_budget", ...)` before any provider call. A `ThinkingMode` value is never a budget and passes the gate. Sonnet 4.6 / Haiku 4.5 keep both flags (their full `_SUPPORTS`), so budget thinking still works there. The same split applies to llama.cpp: `LlamaCppServerProvider` declares `"thinking"` (mapped to the `enable_thinking` chat-template kwarg) but not `"thinking_budget"`, so a `ThinkingMode` works while an int budget fails the gate — see **Thinking control** under the llama.cpp provider quirks.

### Conversation lifecycle

- Construction: `model.new_conversation(name=..., system_blocks=..., tools=..., log_dir=..., log_path=..., **defaults)`. Everything is set here, immutably. `name` defaults to `convo-<8hex>` and doubles as the log filename.
- `add_user_message` / `add_assistant_message` / `add_tool_result` mutate history; `send` / `asend` / `stream` / `astream` are strict single round-trips.
- Tool calls in a response are returned to the caller. Dispatch them yourself (or via `helpers.run_bound_tools`) and append results before the next call.
- `snapshot()` returns an opaque token capturing history; `rollback(snap)` restores it. `clone(*, name=None, log_dir=None, log_path=None, cache_dir=None, cache_mode=None)` deep-copies everything into a fresh conversation that resolves its log path and response-cache settings through the same cascade as a fresh `new_conversation`.

### Provider quirks

- **Anthropic**: System messages are extracted into the separate `system` parameter; `cache=True` on a `SystemBlock` emits `cache_control: ephemeral`. `auto_cache_last_user=True` adds the same marker to the last user block. `cache_ttl` (an `EphemeralCacheTTL` value or `"5m"` / `"1h"`) controls the TTL on every emitted cache_control block; default is the API default (5m). Uses `stop_sequences` (not `stop`). Exports an `AnthropicModel` enum (`from llmfacade.providers.anthropic import AnthropicModel`) listing the current generation only — `OPUS_4_8`, `SONNET_4_6`, `HAIKU_4_5` — each carrying canonical model id, capability metadata, and optional model-scope generation `defaults`. Passing a member to `new_model` auto-applies all of them. `OPUS_4_8` carries a narrowed capability set (`_OPUS_4_8_SUPPORTS = _SUPPORTS - {temperature, top_p, top_k, thinking_budget}`): the Opus 4.7+ API rejects the sampling knobs and the budget-based thinking form with a 400. The sampling knobs are dropped outright (fail fast as `UnsupportedFeature` at construction). The `thinking` knob is **kept** because Opus 4.8 supports adaptive thinking (`thinking=ThinkingMode.ADAPTIVE`); only the pure `"thinking_budget"` flag is dropped, so a budget *value* fails fast at request time while an adaptive mode works (see **Thinking modes** under **Capability gating**). `effort` controls reasoning depth, and `OPUS_4_8.defaults` sets `effort=EffortLevel.XHIGH` + `thinking=ThinkingMode.ADAPTIVE` as model-scope defaults (its recommended settings) — `new_model` applies them for any knob the caller left unset, so a bare `new_model(AnthropicModel.OPUS_4_8)` is pre-tuned; an explicit `effort=`/`thinking=` wins, and `thinking=ThinkingMode.DISABLED` turns thinking off. Passing a raw string opts out of capabilities *and* defaults: full SUPPORTS is used and the caller is responsible for `capability_override=` if needed (e.g. for deprecated 3.x models that lack `thinking`). An explicit `capability_override=` always wins over an enum's default. The enum is a per-release snapshot — bump it when adding new generations.
- **OpenAI**: Talks to the Chat Completions API. Accepts `org_id` as a constructor identity arg (passed to the SDK client). Emits the `max_tokens` knob as the wire param `max_completion_tokens` (the legacy `max_tokens` 400s on the GPT-5 / o-series reasoning models; `max_completion_tokens` is accepted across current models). Supports `effort` → `reasoning_effort` (OpenAI accepts `none`/`minimal`/`low`/`medium`/`high`/`xhigh` — **no `max`**, unlike Anthropic; passed verbatim, so a non-reasoning model or an out-of-range value is the caller's responsibility). `output_format` covers both JSON mode (`OutputFormat.JSON` / `"json"` → `response_format={"type":"json_object"}`) and **strict Structured Outputs**: pass a JSON-Schema `dict` (a bare schema, or a `{name, schema, strict}` config) to emit `response_format={"type":"json_schema", ...}` with `strict=True` by default — the schema must satisfy OpenAI's strict rules (`additionalProperties:false`, every property `required`). Does not currently expose `thinking` / `top_k`, and uses the Chat Completions surface rather than the newer Responses API (see **Future work**).
- **Google (Gemini)**: Converts `"assistant"` role to `"model"`; system messages go to `system_instruction`. Registered under both `"google"` and `"gemini"`. Tool result `function_response.name` is taken from the stored `name` on the `ToolResultBlock` (or looked up via the prior `ToolUseBlock.id` map).
- **llama.cpp (`llamacpp`)**: Talks to a `llama-server` (or `llama-swap`) over its OpenAI-compat HTTP endpoint. No auth. Two modes, decided by the presence of `base_url` at provider construction:
  - **External** (`base_url=...`): identical to phase-1 behaviour. `base_url` points at the OpenAI-compat root, e.g. `http://localhost:8080/v1`; the introspection side channel strips a trailing `/v1` to derive the bare server URL. Server-launch knobs (KV cache quantization via `--cache-type-k/v`, slot persistence via `--slot-save-path`, context size via `--ctx-size`, GPU offload) are CLI flags on `llama-server`, not facade settings — to vary them, run multiple servers (or put llama-swap in front). Passing any `LAUNCH_KNOBS` value (or `name=`) here raises `UnsupportedFeature`.
  - **Managed** (no `base_url`): the provider owns a `llama-swap` subprocess (lazily spawned on first `send`) and the YAML it consumes. `new_model(gguf=..., name=...)` registers a launch entry; per-model launch knobs cascade `provider < model`. `LAUNCH_KNOBS` (defined in `settings.py` alongside `RUNTIME_KNOBS`): `gguf`, `context_size`, `cache_type_k`, `cache_type_v`, `n_gpu_layers`, `n_cpu_moe`, `parallel`, `slot_save_path`, `ttl`, `extra_args`, `fit`, `fit_target`, `fit_ctx`, `flash_attn`, `mmproj_path`, `jinja`, `no_mmap`, `mlock`. Provider-level constructor accepts `llmfacade_dir=` (default `./.llmfacade/`) and `default_ttl=`. Model id is `<gguf-stem>-<hash8>` (canonical-JSON of the launch config) or the explicit `name=`. Adds llama-swap-native methods (sync + async): `running()`, `unload(model_id)`, `unload_all()`, plus `shutdown()` for explicit teardown and `interrupt()` for instant cross-thread abort (see **Instant abort** below). Shutdown defense in depth: OS kill-on-parent-death (Win32 Job Object / Linux `prctl(PR_SET_PDEATHSIG)`), `atexit`+signal handlers, and a PID-file sweep on the next start. The `<llmfacade_dir>` layout is `swap.yaml` (regenerated when entries change), `swap.pid` (PID + port + session UUID), `logs/llamacpp-swap.log`.
  - **Instant abort (`interrupt()`, managed mode only)**: `LlamaCppServerProvider.interrupt() -> bool` hard-kills the running llama-swap backend process tree *immediately* so a thread parked inside a blocking `send()`/`stream()` (i.e. inside the OpenAI SDK's HTTP call) gets a transport error and unblocks — the cross-thread cancel hook a responsive consumer needs. It is **not** a graceful drain: unlike `shutdown()` (which `terminate()`s and waits up to 10s) and unlike the HTTP `unload_all()` (a graceful llama-swap drain), `interrupt()` force-terminates via `_swap_lifecycle._hard_kill_tree` — `taskkill /PID <pid> /T /F` on Windows (whole tree), `proc.kill()` (SIGKILL) on POSIX. It never `killpg`s (the supervisor spawns llama-swap in the parent's process group, so a group kill would take the host down; POSIX orphaned-backend cleanup is a known gap, Windows tree-kill is clean). Thread-safe (takes the supervisor's RLock, which a blocked worker does not hold) and idempotent. **Recoverable**: it resets the supervisor to the not-started state without consuming `_shutdown_done`, so the next `send()`/`stream()` lazily respawns llama-swap and reloads the model via `_ensure_supervised` (rebuilding the OpenAI/httpx clients against the freshly allocated port). Returns `True` if a backend was actually killed, `False` if nothing was running (idle / no model loaded yet) or in **external mode** (we don't own the process there) — never raises for the nothing-to-kill case. Consumer: MTGAI's `interrupt_local_inference()` calls this from an `ai_lock` cancel hook. For the unblock to be *instant*, `_build_clients` constructs the managed-mode OpenAI/AsyncOpenAI clients with `max_retries=0` — otherwise the SDK default (`max_retries=2`) retries twice against the now-dead local port (backoff + connect timeout each) before raising, so the blocked `send()` takes seconds to return after the kill. Managed-mode retries add no value anyway (a localhost backend we deliberately killed won't recover on retry); external mode (a real remote server) keeps the SDK default.
  - **MoE-aware partial offload (`n_cpu_moe`, managed mode only)**: thin passthrough for llama-server's `--n-cpu-moe N` flag. Keeps the **expert weights** of the first N layers on CPU even when those layers' non-expert weights (router + attention + LayerNorms) are on GPU; combine with `n_gpu_layers=-1` to leave the per-token hot path on GPU while burying most expert weights in system RAM. Motivating case: a 22 GB sparse-MoE model (Qwen 3.6 35B-A3B, Gemma 4 26B-A4B) on a 12 GB card — plain `n_gpu_layers=25` puts ~70% of every token's compute on CPU regardless of which experts the router picks; `n_gpu_layers=-1` + `n_cpu_moe=N` keeps only the chosen experts moving across PCIe per token. Picking `N` is manual (depends on per-expert weight size, VRAM budget, active experts per token); facade-level auto-placement is future work. Forwarded to `llama-fit-params` so VRAM estimates account for the chosen split. IS in the `<gguf-stem>-<hash8>` derivation (toggling it yields a different model_id, like `cache_type_k`).
  - **Vision (`mmproj_path`, managed mode only)**: thin passthrough for llama-server's `--mmproj <path>` flag, which loads the multimodal projector alongside the main model. Required to make a vision-capable GGUF (LLaVA, MiniCPM-V, Qwen2-VL, Gemma 3, Qwen 3.6 VL, etc.) actually serve images — without `--mmproj` llama-server falls back to text-only even when the GGUF metadata advertises a multimodal architecture. The wire-format side is already in place: `ImageBlock` content on a user message is emitted as the OpenAI-shaped `{"type": "image_url", "image_url": {"url": "data:<mime>;base64,..."}}` block llama-server consumes (assistant-role images are dropped on the way out, mirroring the rest of the OpenAI-compat surface). The `mmproj_path` file's existence is checked at `new_model()` time the same way `gguf` is. Forwarded to `llama-fit-params` so VRAM estimates account for the projector. IS in the `<gguf-stem>-<hash8>` derivation (toggling it yields a different model_id, like `cache_type_k`) — `mmproj_path` materially changes what the model can do (text-only vs vision). The provider declares the generic `"vision"` capability flag (see **Vision** under **Capability gating**); a GGUF you register without an `mmproj_path` is still nominally `is_available("vision")` at the facade level, so narrow it with `capability_override=provider.SUPPORTS - {"vision"}` if you want the facade to reject image inputs before they reach a text-only server.
  - **Flash attention (`flash_attn`, managed mode only)**: thin passthrough for llama-server's `-fa, --flash-attn [on|off|auto]` flag. `None` (the default) omits the flag entirely so llama-server's `auto` heuristic decides; `"on"` / `"off"` / `"auto"` are validated up front and forwarded verbatim. Quantized V cache (`cache_type_v` of any quant type) hard-couples flash on inside llama-server — passing `flash_attn="off"` against a quantized V cache will cause the server to refuse to start. The motivating case is f16 KV on Gemma 4, where `auto` silently disables flash and costs ~2× wall time; explicit `"on"` reclaims it. Forwarded to `llama-fit-params` so VRAM estimates account for the chosen mode. IS in the `<gguf-stem>-<hash8>` derivation (toggling it yields a different model_id, like `cache_type_k`).
  - **Memory residency (`no_mmap` / `mlock`, managed mode only)**: two boolean launch flags (default `False`, like `jinja`) that tune *where the model's bytes live* without changing what it emits — the motivating case is a low-VRAM sparse MoE with expert weights spilled to system RAM (see `docs/learnings/llamacpp-low-vram-moe-flags.md`). `no_mmap=True` → `--no-mmap`: reads the whole model into RAM up front instead of mmap'ing it, so the default demand-paging of experts from disk mid-token disappears (measured +35% t/s in the source benchmark). It's **opt-in only** because it forces a full preload — the model must fit in RAM, or it backfires / fails to load on a tight-RAM box. `mlock=True` → `--mlock`: pins the (already-RAM-resident) pages so the kernel can't page them back to disk under memory pressure/idle — not a speed flag, it's the fix for the long-running-server "day-3 slowdown" (directly relevant since managed mode is a long-lived supervised backend). Under Docker `--mlock` silently no-ops unless the container raises the `memlock` ulimit **and** is granted the `IPC_LOCK` capability (no error is raised — it just leaks back to default paging). Both are **excluded** from the `<gguf-stem>-<hash8>` derivation (memory residency ≠ generation behaviour, so toggling either mustn't shift `model_id` or break slot-cache continuity — same treatment as `fit`), and both are **NOT forwarded** to `llama-fit-params` (they don't affect the VRAM fit, and `--no-mmap` would force the probe to preload the whole model, blowing its sub-second/15s-capped budget).
  - **Autofit (`fit` / `fit_target` / `fit_ctx`, managed mode only)**: `fit` defaults to `True` and bakes `--fit on` into the rendered swap.yaml so newly registered models are OOM-safe by default — llama-server adjusts unset launch args to fit available VRAM at spawn time. `fit_target` is a tuple of per-device free-margin MiB targets (rendered as a comma-joined `--fit-target`); `fit_ctx` is the minimum context-size floor that `--fit` won't shrink past. The fit knobs are **excluded** from the `<gguf-stem>-<hash8>` derivation so flipping `--fit on/off` doesn't shift `model_id` and break slot-cache continuity. Each `new_model()` also runs `llama-fit-params` once (if the binary is on PATH and `fit=True`), parses the printed estimate, and stashes it on the provider keyed by `model_id`. The probe is synchronous — `new_model()` blocks for at most `_FIT_PARAMS_TIMEOUT_SECONDS` (15s; sub-second on a warm GPU) and `extra_args` are intentionally NOT forwarded (those are llama-server flags that fit-params would reject). The conversation's JSONL settings header (and HTML log) then gains a top-level `fit_estimate` block of `{context_size, n_gpu_layers, parallel, est_vram_mib}` via the new `Provider.log_metadata(model_id=...)` hook. The block is labelled an *estimate*, not wire-truth — llama-server re-fits at spawn against current VRAM, which can drift between registration and first send. Subprocess failures (binary missing, non-zero exit, timeout, unparseable output) silently store `None` so logging never blocks `new_model()`.
  - **Thinking control (`thinking` knob + `jinja`)**: the `thinking` runtime knob is in `SUPPORTS` and maps a `ThinkingMode` to llama.cpp's `chat_template_kwargs` via `extra_body` — `ADAPTIVE`/`ADAPTIVE_SUMMARIZED` → `{"enable_thinking": true}` (no "summarized" display mode exists locally, so both mean thinking-on), `DISABLED` → `{"enable_thinking": false}`. Mirrors Anthropic's `_thinking_to_api`: one uniform knob value, a provider-specific wire form. llamacpp declares `"thinking"` but **not** `"thinking_budget"`, so an int token budget fails fast through the value-level gate in `Conversation._build_request` (the same mechanism that rejects a budget on Opus 4.8) instead of being silently dropped. The `enable_thinking` kwarg is only honored when the GGUF's embedded chat template gates reasoning that way (Gemma 4 / Qwen3) **and only under `--jinja`** — hence the managed-mode `jinja` launch knob (default `True`), which bakes `--jinja` into the rendered swap.yaml so llama-server renders the embedded template instead of its built-in format detection. `jinja` IS in the `<gguf-stem>-<hash8>` derivation (toggling it swaps the chat template, materially changing output, like `cache_type_k`); it is **not** forwarded to `llama-fit-params` (it doesn't affect VRAM and fit-params would reject the flag). `--jinja` is also a prerequisite for correct tool-calling on newer Gemma 4 / Qwen3 quants whose embedded template is the only correct one. Do **not** force `tool_choice` with thinking on — llama.cpp misroutes the tool call into `reasoning_content` (ggml-org/llama.cpp#20809); use `tool_choice="auto"` plus a retry-if-no-toolcall guard. See `docs/learnings/llamacpp-reasoning-tool-calling.md`.
  - **Thinking-style auto-detect (`thinking_style`, managed mode)**: at `new_model()` the provider reads the GGUF's embedded `tokenizer.chat_template` (stdlib header parse, no weights — `providers/_gguf.py`) and classifies it into a `ThinkingStyle` (`TEMPLATE_KWARG` / `REASONING_BUDGET` / `THINK_TOKEN` / `DEFAULT` / `UNKNOWN`). An explicit `thinking_style=` (a `ThinkingStyle` or its validated string value) always wins; external mode has no local GGUF so the style is `UNKNOWN` unless set explicitly (then stored). The style is kept per `model_id` (like `_fit_estimates`) and surfaced in the conversation log header via `log_metadata`. Its only behavioural effect: when `thinking` is set against a model whose style is a recognised non-kwarg style (anything but `TEMPLATE_KWARG`/`UNKNOWN`), `_build_kwargs` warns **once** per `(model_id, style)` that the knob likely has no effect — the kwarg is still emitted (harmless), never silently dropped. The `enable_thinking` translation itself is style-agnostic; detection only drives the warning and the log.
  - **Both modes**: uses the OpenAI Python SDK for chat transport (so `tool_choice` works the same as on OpenAI), routing the llama.cpp-specific samplers `top_k` / `min_p` / `repeat_penalty` through the SDK's `extra_body=` argument verbatim.
  - **DRY sampler (`dry` knob, both modes)**: llama.cpp's DRY ("Don't Repeat Yourself") sampler is an *n-gram-level* repetition penalty — the tool for paragraph/verbatim loops that token-level `repeat_penalty` can't break. Exposed as a single structured runtime knob `dry` whose value is a `DrySampler` frozen dataclass (`from llmfacade import DrySampler`): `multiplier` (required; the enabling param, `0.0` disables), `base=1.75`, `allowed_length=2`, `penalty_last_n=-1`, `sequence_breakers: tuple[str,...] | None = None`. `LlamaCppServerProvider._dry_to_extra_body` unpacks it into the wire params `dry_multiplier` / `dry_base` / `dry_allowed_length` / `dry_penalty_last_n` / `dry_sequence_breakers` on `extra_body` (same channel as the samplers above), omitting any `None` field so the server keeps its own default (only `sequence_breakers` is omittable). Only `llamacpp` declares `"dry"` in `SUPPORTS`; setting it on any other provider raises `UnsupportedFeature` (or is dropped-with-warning if it cascades down), exactly like `repeat_penalty`. The whole DRY config is a unit — a per-call `dry=` wholly replaces a higher-scope default (no per-field merge), which suits the "escalate to anti-loop sampling on a retry" use case. It lands in the response-cache fingerprint via the normal merged-settings path (`cache._normalize` and the log's `_logsafe` both unwrap the dataclass to a plain dict). Introspection methods (sync + async): `health()`, `slots()`, `save_slot(id_slot, filename)`, `restore_slot(id_slot, filename)`, `erase_slot(id_slot)`. `count_tokens()` calls `POST /tokenize` and falls back to `chars/4` on connection error. Supports first-class `min_p`. Does NOT advertise `context_size` or `keep_alive` as runtime knobs — both are server-launch concerns and live in `LAUNCH_KNOBS` (managed mode only).
  - **Reasoning capture (both modes)**: llama-server surfaces a reasoning model's extracted chain-of-thought as `reasoning_content` (DeepSeek convention; the default when `--reasoning-format` parsing is on) on the message (non-stream) and `delta.reasoning_content` (stream). `_parse_response` turns it into a leading `ThinkingBlock` (so reasoning shows in the log and round-trips in history); the stream path emits it as `thinking_delta`s and flushes one consolidated `ThinkingBlock` before the first text. `reasoning_tokens` is read from `completion_tokens_details.reasoning_tokens` if the build reports it (most don't → `0`; the conversation log then counts the reasoning text locally via `/tokenize`). The module-level `_llama_reasoning` / `_llama_reasoning_tokens` helpers also tolerate the `reasoning` field some OpenAI-compat builds use. Note the assistant-side `ThinkingBlock` is still dropped on the *outbound* wire (`_message_to_api`) — llama-server has no canonical thinking-block input format.
  - **Introspection routing in managed mode**: llama-swap does not proxy llama-server-native paths via the bare URL, but it does expose `/upstream/<model_id>/<arbitrary-path>` (with on-demand model load), so we route through that. Every per-backend introspection method on the provider (`health`, `slots`, `save_slot`, `restore_slot`, `erase_slot`, plus `count_tokens` via its existing `model_id=`) takes an optional `model: str | None = None` kwarg. In managed mode: `model` is required when more than one entry is registered; with exactly one entry the resolver infers it; URL-quotes slashes (e.g. `Qwen/Qwen2.5-3B`) so llama-swap parses the model id correctly. In external mode the `model` kwarg is silently ignored — bare `llama-server` has no model routing. `Model` exposes mirror methods (`model.health()`, `model.slots()`, `model.save_slot(...)`, etc., sync + async) that auto-bind `self._model_id` — same precedent as `Model.count_tokens()`. Special case: `provider.health()` with no `model=` in managed mode hits the swap's own `/health` (which returns plain text `"OK"`) and normalises the result to `{"status": "ok"}`; pass `model=` (or use `Model.health()`) to get the per-backend JSON. `count_tokens` silently falls back to chars/4 if the routing helper raises (no models registered) so cache-summary logging never blocks. `running()` / `unload()` / `unload_all()` are llama-swap-native and don't take `model=`.
  - **Conversation slot save/restore (external mode only)**: `Conversation.save_slot(filename)` / `restore_slot(filename)` / `erase_slot()` (sync + async) wrap the raw provider-level `/slots/{id}?action=...` calls so callers don't have to encode the five undocumented quirks of the endpoint. `warm_and_save(filename, *, max_warmup_tokens=1)` is the canonical "warm a static-prefix KV and persist it" recipe: it takes a snapshot, drives a one-token completion against the conversation's current system block, rolls history back, and saves slot 0. All four require `--slot-save-path` to be set on llama-server; if not, the wrapped `ProviderError` includes a hint pointing at the missing flag. Filenames are sanitised against `[a-zA-Z0-9._-]+` (no `..`, no separators, no leading dot) so server-side `fs_validate_filename` rejections are caught client-side. v1 is hardcoded to slot 0 — fine for the canonical `--parallel 1` deploy. Managed mode raises `NotImplementedError`: slot semantics across llama-swap model loads are undefined (KV state is per-architecture and the swap may evict the model between calls); callers who really want this can still call `provider.save_slot(model=<id>, ...)` directly. Atomicity: the conversation methods do not internally lock — wrap a `restore_slot` + `send` pair (or `save_slot` + `send` for warmup composition) with `with provider.slot_lock(): ...` (or `async with provider.aslot_lock(): ...`) when concurrent tasks may mutate slot state. `slot_lock()` / `aslot_lock()` return the bare `threading.Lock` / `asyncio.Lock` instances (per-provider, initialised in `_init_client`); the two locks do not synchronise across the sync/async boundary, so a process should pick one path. The `?action=erase` endpoint clears slot 0's in-memory KV only — it does NOT delete files under `--slot-save-path`. Callers who need that do filesystem-level cleanup themselves; `--slot-save-path` accumulates `.bin` files indefinitely otherwise.
- **localimage**: Local image generation against an OpenAI-compatible image server (stable-diffusion.cpp's `sd-server` — the image analog of `llama-server`, supporting Flux / SD / SDXL / Qwen-Image — or LocalAI). Image-only provider (no chat hooks); `SUPPORTS = {"image_generation"}`. **External mode only** for now: `base_url` is required at construction (raises `LLMError` otherwise) and points at a running server, e.g. `LLM().new_provider("localimage", base_url="http://127.0.0.1:1234/v1")`; no auth (a placeholder `api_key` satisfies the SDK). Reuses the OpenAI Images transport (`/v1/images/generations` + `/v1/images/edits`) via the `openai` SDK — the same reuse trick llamacpp uses for chat — and always requests `response_format="b64_json"` (local servers default to returning URLs). Server-specific knobs (steps, cfg, sampler, seed) go through `extra=` — forwarded as the SDK `extra_body` for LocalAI; for `sd-server` use its `<sd_cpp_extra_args>{...}</sd_cpp_extra_args>` prompt syntax. A **managed mode** that spawns/supervises `sd-server` (mirroring the llamacpp `llama-swap` supervisor) is planned — see **Future work**.

### Image generation

A separate one-shot surface from the chat/`Conversation` flow — image generation does not go through `CompletionRequest` or the settings cascade. The hierarchy mirrors the text side:

- `provider.generate_image(prompt, *, model=None, n=1, size=None, aspect_ratio=None, quality=None, background=None, output_format=None, reference_images=None, save_dir=None, extra=None)` (plus async `agenerate_image`) returns an `ImageResult`.
- `provider.new_image_model(model_id, **defaults)` returns an `ImageModel`; `image_model.generate(prompt, **overrides)` / `agenerate` apply the per-model defaults (a per-call value wins) and forward.
- `LLM.generate_image(prompt, *, provider, model, base_url=None, api_key=None, **gen_kwargs)` (plus async `agenerate_image`) is the single entry point: it resolves the provider from `PROVIDER_REGISTRY` (hosted providers use the manager `api_keys`/env; `localimage` needs `base_url`) and caches it per `(provider, base_url)`.

`ImageResult` carries `images: list[ImageBlock]` (the generated bytes, so they round-trip straight back into a vision request), `usage: ImageUsage | None`, `model`, `provider`, `paths` (filled only when `save_dir=` was passed), and `raw`. `ImageResult.save(dest, *, prefix="image")` writes the images into a directory as `<prefix>_<i><ext>` (extension derived from each block's `media_type`). `ImageUsage` carries `input_tokens` / `output_tokens` / `total_tokens` / `image_count`; **no provider returns a dollar figure** — derive cost from the token counts and published pricing. The token fields are populated only where the provider breaks them out (OpenAI `gpt-image-*`, Gemini-native `usage_metadata`); `image_count` is always set.

**Capability gating.** `"image_generation"` is a pure capability flag in `SUPPORTS` (like `"vision"`), declared by OpenAI, Google and localimage; query via `is_available("image_generation")` / `get_capabilities()`. Providers without it (Anthropic, llamacpp) inherit the base `generate_image`, which raises `UnsupportedFeature("image_generation", ...)`. `ImageModel` narrows via `capability_override`, like `Model`.

**Reference images** (image-conditioning — keep a character/subject on-model, edit, blend) are passed as `reference_images: list[ImageBlock]`:
- **OpenAI** routes them to `images.edit` (up to 16 references, uploaded as `(filename, bytes, mimetype)` tuples); a `mask` for inpainting goes via `extra={"mask": <file/bytes>}`. Without references it uses `images.generate`. Default model `gpt-image-1`; reads `data[i].b64_json` and does not send `response_format` (which `gpt-image-*` rejects).
- **Google** is **Gemini-native only** (`gemini-2.5-flash-image`, "Nano Banana") via `generate_content` with `response_modalities=["IMAGE"]`; reference images are appended to `contents` as `inline_data` parts (the same wire shape as vision input), and `aspect_ratio` maps to `image_config.aspect_ratio`. **Imagen is intentionally not supported** — it is being shut down per Google's Gemini API deprecations, and on the api-key Developer API it could not take reference images anyway (that path was Vertex-only).
- **localimage** routes references to the server's `/v1/images/edits` (img2img), same as OpenAI.

The OpenAI-Images request-building and response-parsing is shared by the OpenAI and localimage providers in `providers/_openai_images.py` (`build_generate_kwargs` / `build_edit_kwargs` / `parse_images_response` / `media_type_for`); the only behavioural difference is `request_b64` (localimage requests base64; OpenAI omits the param).

**Image-generation logging (audit ledger).** Image generation does not flow through the conversation JSONL/HTML turn format (it has no turns, history, system blocks or tools). Instead every successful `generate_image` / `agenerate_image` call appends one record to a **shared ledger** in the manager's run dir — `<run_dir>/images.jsonl` (the programmatic source of truth) and a human-readable `<run_dir>/images.html` sibling (header written once, then one `<section>` per call). The motivation is *spend*: hosted image generation costs real money, so the ledger is an audit trail of prompt, model, provider, `n`, request params, `ImageUsage` token counts, per-image `{media_type, bytes}`, and any saved `save_dir` paths — **never** the image bytes, and (for references) only a *count*. The hook lives in `Provider.generate_image` / `agenerate_image` (the template methods), so all three entry points — direct, `ImageModel.generate`, `LLM.generate_image` — are logged exactly once. Logging reuses the existing **`log_dir` cascade switch** (no new knob): on by default, disabled by `log_dir=False` at the manager or provider scope, redirected by an explicit `log_dir=<dir>` on the provider. There is no model/convo layer for one-shot images, so the cascade is just provider-then-manager (`resolve_image_log_path`). Writes are **best-effort**: `log_image_generation` swallows any error and warns once, so a logging failure never prevents returning the paid-for image. Implementation in `_image_log.py`. Only *successful* generations are logged (a failed call propagates its exception and generally isn't billed).

## Adding a new provider

1. Create `src/llmfacade/providers/<name>.py` with a `class <Name>Provider(Provider)`.
2. Set `NAME`, `API_KEY_ENV`, and a `SUPPORTS: frozenset[str]` of knob names.
3. Implement `_init_client()` and `_complete_raw` / `_acomplete_raw` / `_stream_raw` / `_astream_raw`. Each receives a `CompletionRequest` (model, messages, system_blocks, tools, stop, plus a single merged `settings: dict[str, Any]` and `settings_source: dict[str, str]`). Generation knobs — including `tool_choice` — live in `settings`; read them via `req.settings.get("tool_choice", "auto")`.
4. Register `(module_path, class_name)` in `PROVIDER_REGISTRY`.
5. Add an optional dependency block to `pyproject.toml`.
6. Map SDK errors to facade exceptions (`AuthenticationError`, `RateLimitError`, `ProviderError`).

If your provider needs extra identity args (e.g. `org_id`), add them to a `__init__` override that pops them before calling `super().__init__(**kwargs)`. The base accepts `manager`, `api_key`, `base_url`, plus the 16 generation kwargs (gated by `SUPPORTS`).

Optional hook: override `Provider.log_metadata(*, model_id) -> dict[str, Any] | None` to surface provider-specific extras (a fit estimate, backend version, etc.) as siblings of `settings` in the conversation's JSONL log header and HTML log. Default returns `None`.

`SUPPORTS` carries six pure capability flags alongside the runtime knobs: `"tools"` (the provider can route a tool list — without it, `Conversation(tools=[...])` raises `UnsupportedFeature` at construction), `"tool_choice"` (forced selection beyond `"auto"` is supported), `"vision"` (the provider can route a top-level `ImageBlock`), `"tool_result_images"` (the provider can route an `ImageBlock` returned inside a `ToolResultBlock` — Anthropic only), and `"thinking_budget"` (the provider/model accepts legacy budget-based extended thinking — see **Thinking modes** under **Capability gating**). See **Vision** under the **Capability gating** section above for the image flags. All five are queryable via `is_available(...)` / `get_capabilities()` and narrowable per-model via `capability_override`. The flags are orthogonal: a provider may declare `"tools"` without `"tool_choice"` if its API has no forced-selection mode, `"vision"` without `"tool_result_images"`, or `"thinking"` (the knob) without `"thinking_budget"` (e.g. Opus 4.8, which supports only adaptive thinking). For models that don't support a feature, callers narrow with e.g. `capability_override=provider.SUPPORTS - {"tools"}` (or `- {"vision"}`) on `new_model()`. The sixth flag, `"image_generation"`, gates the separate `generate_image` surface (see **Image generation**) and is declared by OpenAI, Google and localimage.

## Code Style

- Ruff with rules `E, F, I, UP, B, SIM`; line length 99.
- All wire-format models use `@dataclass(frozen=True, slots=True)`.
- Full type annotations throughout.
- src/ layout - imports as `from llmfacade import ...`.
- Naming: snake_case throughout. Public lifecycle methods (`new_provider`, `new_model`, `new_conversation`, `send`, `stream`, `snapshot`, `rollback`, `clone`, `add_user_message`, etc.) and helper module functions are all snake_case. Class names and dataclass field names follow the usual Python conventions.

- **Managed local image generation (`localimage`).** The `localimage` provider currently supports external mode only (point `base_url` at a running OpenAI-compatible image server). A managed mode that spawns and supervises a `sd-server` (stable-diffusion.cpp) subprocess — mirroring the llamacpp `llama-swap` supervisor (model registry, PID file, OS kill-on-parent-death, lazy spawn on first call) — would make local Flux/SD generation as turnkey as managed-mode llama.cpp. Deferred to keep the first image-generation PR reviewable; the external-mode surface is forward-compatible (managed mode adds `new_image_model(gguf=...)`-style registration without changing `generate_image`).
- **Vision-projector autodetection (llamacpp managed mode).** Scanning the GGUF directory for a sibling `mmproj-*.gguf` and auto-binding it would save callers a path; punted for now in favour of the explicit `mmproj_path=` knob, which is unambiguous and debuggable. Worth picking up once the registered set of vision models is large enough that boilerplate dominates.
- **Thinking-style auto-detect in external mode (llamacpp).** Managed mode reads the GGUF's embedded chat template at `new_model()` to set `thinking_style`; external mode has no local file, so the style is `UNKNOWN` unless the caller passes `thinking_style=`. llama-server exposes the rendered template via `GET /props`, so external mode *could* detect it — but that needs a live-server network call at `new_model()` time, which today does zero I/O in external mode. Deferred to preserve that property; revisit if external-mode thinking-control ergonomics matter.
- **Google `cachedContents` integration.** Gemini supports prompt caching but via a separate resource API: `client.caches.create(...)` returns a named cache handle that subsequent `generate_content` calls reference via `cached_content=<handle>`. This is a caller-managed lifecycle (create/list/delete + TTL) rather than a per-request `cache_control` marker, so the Anthropic-style `auto_cache_tools` / `auto_cache_last_user` knobs don't translate. Designing a facade-level surface for it (probably a `Provider.create_cache(...)` returning a handle, plus a `cached_content=` argument on `Conversation`) is its own piece of work and is deferred. Until it lands, callers running tool-heavy workloads against Gemini cannot get the equivalent of Anthropic's tool-array caching.
- **OpenAI / llamacpp tool caching — intentionally not implemented.** OpenAI auto-caches prompt prefixes above ~1024 tokens transparently (no per-request marker, no knob to flip), and llama-server's prompt cache is fully internal (the server reuses overlapping prefixes per slot and there's no client-facing `auto_cache_tools` semantic to expose). Adding `auto_cache_tools` to either would be a misleading no-op. This is documented here so the absence is understood as deliberate, not an oversight.
- **OpenAI Responses API.** The OpenAI provider targets Chat Completions. OpenAI now recommends the Responses API (`client.responses.create`) for new work — it round-trips reasoning items (we currently drop `ThinkingBlock`s outbound, since Chat Completions can't carry them) and reports materially better prompt-cache utilization on reasoning models. Migrating is a provider-internal swap (the facade surface wouldn't change) but non-trivial — different request/response item shapes, `text.format` instead of `response_format`, streaming event differences — so it's deferred. Chat Completions remains fully supported in the meantime.
- **`OpenAIModel` capability enum.** Unlike Anthropic, the OpenAI provider takes raw string model ids with no per-model capability narrowing, so setting `temperature`/`top_p` on a GPT-5 reasoning model (which 400s — only `temperature=1` is accepted) isn't caught up front the way the Anthropic enum catches it. An `OpenAIModel` enum mirroring `AnthropicModel` (narrowing sampling knobs and defaulting `reasoning_effort` for known reasoning models) would close that gap; deferred because OpenAI's model lineup churns faster and raw strings already work.
